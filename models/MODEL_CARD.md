> 배포 중인 모델을 **추적 가능**하게 유지하기 위한 모델 카드. PRD §FR-3 / §6.2 "모델 버전 관리(규칙 확정)"의 산출물.
> 학습(Colab GPU) 완료 시마다 아래 "버전 이력"에 **한 행씩 append**한다. 현재는 온보딩(2026-07-15)·1차 학습 이전이라 **규칙 정의 + 값 미확정(TBD)** 상태.

## 1. 파일명·버전 규칙 (확정)

- **가중치 파일명**: `{날짜}_{데이터셋버전}_{mAP}`
  - 예: `20260710_v1_0.71` (2026-07-10 학습 · 데이터셋 v1 · mAP@0.5 0.71)
  - `{날짜}`=`YYYYMMDD`, `{데이터셋버전}`=`v1`·`v2`…, `{mAP}`=`mAP@0.5` 소수 둘째자리
- **MLflow run 이름**: `{날짜}_{데이터셋버전}` — `ai-dl/tracking.py`의 `_run_name()`과 **동일 규칙**(mAP는 run 이름엔 없고 메트릭으로 기록). 파일명은 여기에 `_{mAP}`만 덧붙인 형태.
- **산출물 배치**: 학습 산출물(`.pt` → ONNX 변환 `.onnx`)은 `models/{파일명}/`에 두고, 이 카드에 해당 버전 행을 추가한다. (가중치 바이너리 자체는 레포에 커밋하지 않음 — Drive/서버 배포본으로 관리)

## 4. ARM CPU 추론 최적화 (설계 확정)

- 학습 산출물에 **ONNX 변환 + INT8 양자화** 파이프라인 포함 → ONNX Runtime 서빙.
- 순정 PyTorch 대비 **2~4배 속도** 확보 (100장 10분 목표의 전제).
- 최적화를 후반으로 미루지 않고 **1차 학습부터 적용**.
- FastAPI 워커 분리: CPU-bound YOLO 추론은 별도 워커 프로세스로 오프로드(3 OCPU: 탐지용 1 + API용 1~2).

## 7. 한계·주의

- 학습 데이터가 공개 데이터셋 중심이라 실제 현장 분포와 차이 가능 → 배포 후 오탐 모니터링·재학습 전제.
- 3종 외 하자(누수·백태·도장 손상)는 탐지 불가 — 검수자 수동 확인 필요.
- 심각도 등급은 모델이 아닌 **규칙 기반**(크기·유형)으로 산정(A~E 5등급) 후 점검자 수동 조정. 상세: `conventions/하자_심각도_등급_규칙.md`(HAJA-109).
- INT8 양자화로 인한 정확도 손실은 1차 학습 시 FP32 대비 mAP 차이로 측정·기재.
