# hajaCheck — 로컬 개발 가이드

> 대상: 기능별 풀스택 담당자(백엔드+프론트를 함께 구현) 전원
> 배경: 전역 인증 강제(`anyRequest().authenticated()`) 도입 이후 로그인 없이는 대부분의 API가 401을 반환하고,
>  User 엔티티가 PostgreSQL named enum(`@JdbcTypeCode(NAMED_ENUM)`)에 매핑돼 있어 H2로는 로컬 기동이 안 된다.
>  이 문서 하나로 "클론 → 로컬 기동 → 로그인 → 내 기능 개발 → PR"까지 끝낸다.
> 연관 문서: `SpringBoot_코드_컨벤션.md`, `React_코드_컨벤션.md`, 루트 `CLAUDE.md`(워크트리 워크플로우)

---

## 1. 개요

- 인증은 **세션 쿠키** 방식이다(JWT 아님). `SecurityConfig`가 `anyRequest().authenticated()`를 걸어놓아서,
  `/api/auth/**`·`/oauth2/**`·Swagger·헬스체크를 제외한 모든 API는 로그인된 세션이 있어야 응답한다.
- 그래서 로컬에서 "내 기능 화면/엔드포인트"를 확인하려면 매번 로그인부터 해야 한다 — 이 문서 6번 섹션의 시드 계정으로 바로 로그인한다.
- 스택: 백엔드 Java 17 · Spring Boot 3.3.5(포트 `8080`) / 프론트 Vite 6 · React 18(포트 `5173`,
  `/api`·`/ai`·`/ws`는 Vite dev proxy로 백엔드·ai-server에 전달) / DB PostgreSQL 16 / 세션 저장소 Redis.

---

## 2. 사전 준비

- JDK 17 (`java -version`으로 확인 — `backend/build.gradle` toolchain 17)
- Node.js (프론트 `package.json` 기준, LTS 권장) + npm
- Docker Desktop (또는 Docker 엔진) — 로컬 PostgreSQL·Redis 컨테이너용
- 레포 클론 후 `backend/`·`frontend/` 각각에서 작업

---

## 3. 로컬 인프라 기동 (PostgreSQL + Redis)

**로컬 DB는 반드시 실제 PostgreSQL이어야 한다 — H2로 대체 불가.** `User` 엔티티의 `role`/`status`/`socialProvider`가
PG named enum 타입(`role_type` 등)에 `@JdbcTypeCode(SqlTypes.NAMED_ENUM)`으로 매핑돼 있어서, H2에는 해당 enum
타입 자체가 없어 스키마 생성이 실패한다. 저장소 루트의 `docker-compose.yml`에 이미 `postgres:16`(기본 DB명
`hajacheck_dev`)·`redis:7` 서비스가 정의돼 있으므로 그 두 개만 띄운다.

```bash
# 루트 .env 가 없다면 .env.example 복사 후 DB_USER/DB_PASSWORD/REDIS_PASSWORD/POSTGRES_DB 채우기
cp .env.example .env

docker compose up -d postgres redis
```

- Postgres는 `127.0.0.1:5432`, Redis는 `127.0.0.1:6380`(컨테이너 내부 6379)에 바인딩된다(둘 다 로컬 전용, 외부 미공개).
- 컨테이너 자체는 매번 새로 만들 필요 없이 한 번 띄워두고 계속 재사용하면 된다(`docker compose stop postgres redis`로 중지,
  `up -d`로 재기동 — 볼륨(`pgdata`/`redisdata`)이 있어 데이터는 유지된다).
- ⚠️ **DB명 주의**: 이 compose의 `POSTGRES_DB` 기본값은 `hajacheck_dev`인데, 4번의 `application-local.yml.example`은
  `hajacheck`로 접속하도록 돼 있다. 아래 둘 중 하나로 맞출 것.
  - (권장) 루트 `.env`에 `POSTGRES_DB=hajacheck`를 명시해 compose가 만드는 DB명을 예시 설정에 맞춘다.
  - 또는 4번에서 복사한 `application-local.yml`의 `spring.datasource.url` 끝을 `hajacheck_dev`로 바꾼다.

## 4. 백엔드 로컬 실행

```bash
cd backend
cp src/main/resources/application-local.yml.example src/main/resources/application-local.yml
# application-local.yml 은 gitignore 대상 — datasource.url/username은 3번에서 띄운 로컬 PG를 그대로 가리킨다.
```

- 시크릿(`DB_PASSWORD`, `REDIS_PASSWORD` 등)은 `application-local.yml`에 평문으로 적지 말고 루트 `.env`에 두고 주입한다.
  - 터미널: `export $(grep -v '^#' ../.env | xargs) && ./gradlew bootRun`
  - IntelliJ: EnvFile 플러그인으로 Run Configuration에 `.env` 연결 후 프로파일 `local`로 실행
- 카카오/구글 OAuth 키(`KAKAO_CLIENT_ID` 등)가 없으면 소셜 로그인 버튼만 동작 안 하고, 나머지 기동에는 영향 없다
  (6번에서 다루듯 로컬 개발은 시드 계정의 자체 로그인으로 충분).
- `ddl-auto: update`이므로 최초 기동 시 enum 타입 3개 + 테이블이 자동 생성된다.
- 기동 로그에서 아래 두 줄을 확인하면 정상이다.
  - Hibernate가 `role_type`/`user_status_type`/`social_provider_type` 등을 만드는 로그(`show-sql: true`)
  - `LocalUserSeeder`가 찍는 `로컬 시드 유저: dev@hajacheck.local / dev1234` (이미 있으면 "생성 스킵" 로그)
- Swagger: `http://localhost:8080/swagger-ui.html` — `/api/auth/login`으로 먼저 로그인 세션을 만들면 나머지
  엔드포인트도 브라우저 세션 그대로 Swagger에서 호출 가능(같은 브라우저·같은 오리진 기준).

## 5. 프론트 로컬 실행

```bash
cd frontend
cp .env.example .env.local   # VITE_KAKAO_MAP_APP_KEY 등, 없어도 지도 관련 기능 외엔 기동에 지장 없음
npm install
npm run dev
```

- `npm run dev`는 **개발 모드에서 MSW(Mock Service Worker)를 항상 켠다**(`src/app/main.tsx`의 `enableMocking()`).
  MSW 핸들러가 등록된 요청(`/api/auth/login`, `/api/users/me`, 그 외 각 feature의 `*.handlers.ts`)은 무조건
  가짜 응답으로 가로채지고, **핸들러가 없는 요청만** Vite proxy를 통해 실 백엔드로 그대로 나간다
  (`onUnhandledRequest: 'bypass'`).
- 즉 현재 구조에서 "MSW 목 모드"와 "실 백엔드 연동 모드"는 자동 전환 스위치가 없다.
  - **화면만 빠르게 만들 때**: 그냥 `npm run dev` — 로그인도 목 계정(`hajacheck` / `password1234`, `authApi.handlers.ts`)으로 통과된다.
  - **실 백엔드로 통합 확인할 때**: `src/app/main.tsx`의 `enableMocking()` 호출을 로컬에서만 임시로 주석 처리하고
    다시 `npm run dev` — 이러면 로그인 포함 모든 요청이 Vite proxy로 실 백엔드에 붙는다(**이 주석 처리는 커밋 금지**).
    본인 담당 화면 확인이 끝나면 원복한다.

## 6. 로그인해서 개발하기

1. 백엔드(4번)가 `local` 프로파일로 떠 있어야 `LocalUserSeeder`가 시드 계정을 만든다.
2. 5번에서 MSW를 끈 상태로 브라우저에서 `http://localhost:5173/login` 접속.
3. 기업 로그인(자체 email/password)에 시드 계정 입력:
   - **email/loginId**: `dev@hajacheck.local`
   - **password**: `dev1234`
4. 로그인 성공 시 세션 쿠키(`SESSION`, Redis 저장)가 발급되고, 이후 같은 브라우저의 모든 요청이 로그인된 사용자로 처리된다.
   시드 계정은 `role=ADMIN`이라 로컬 개발 편의상 권한 제한 없이 대부분의 화면에 접근할 수 있다.
5. API만 단독으로 확인하고 싶으면 Swagger(`/swagger-ui.html`)에서 `/api/auth/login`을 먼저 호출해 같은 세션을 만든 뒤
   나머지 엔드포인트를 그대로 호출한다.
6. 소셜 로그인(카카오/구글)은 로컬에서 OAuth 키가 없으면 동작하지 않는다 — 로컬 개발은 항상 이 기업(자체) 로그인으로 진행한다.

## 7. 기능 풀스택 개발 사이클

1. **프론트 UI 먼저**: 담당 기능의 화면/컴포넌트를 MSW 목 데이터(`features/{기능}/api/*.handlers.ts`)로 붙여 먼저 완성한다.
   화면 레이아웃·상태 흐름은 백엔드가 없어도 확인 가능하다.
2. **백엔드 엔드포인트 구현**: `/api/{기능}`을 `core/{도메인}` 아래 controller/service/repository/dto로 구현한다.
   전역 인증이 이미 `anyRequest().authenticated()`로 걸려 있으므로 **새 엔드포인트에 별도 permitAll 설정을 추가할 필요 없다**
   (공개 API가 필요한 예외적인 경우만 `SecurityConfig`의 permitAll 목록에 명시 추가 — 팀 협의 후).
3. **실 백엔드 통합 확인**: 5번의 방법으로 MSW를 끄고, 6번의 시드 계정으로 로그인한 뒤 실제 화면에서
   본인 엔드포인트가 기대대로 동작하는지 확인한다.

## 8. 로컬 테스트 → 올리기(PR) 플로우

루트 `CLAUDE.md`의 워크트리 워크플로우를 이 레포 기준으로 요약하면:

1. 개별 워크트리·브랜치 생성(`git worktree add ../hajacheck-{역할} -b {역할}/{이슈번호}-{내용}`) — **main/dev 직접 코드 커밋 금지**.
2. 코드 + 테스트 작성. 스택별 빌드/테스트:
   - 백엔드: `./gradlew compileJava` && `./gradlew test`
   - 프론트: `npm run build`(+`npm test`)
3. 커밋 전 시크릿 스캔(값 평문 노출 금지, `.env`류는 항상 gitignore 대상).
4. PR 생성 — **base는 `dev`**(이 레포는 `main` 아님). `Closes #N` 포함.
5. `dev`로의 PR은 **PR머신이 자동 검수·머지**한다 — 사람이 먼저 머지하지 않는다.
6. `dev → main` 승격은 PR머신 검수 후 **사람 핑(hold_promote)** — 자동 머지 아님, 운영자 승인 필요.

## 9. 자주 걸리는 것 (FAQ)

| 증상 | 원인 / 해결 |
|---|---|
| API 호출이 죄다 `401 AUTH_UNAUTHORIZED` | 로그인 세션이 없다. 6번대로 시드 계정(`dev@hajacheck.local` / `dev1234`)으로 먼저 로그인. |
| 백엔드가 H2로는 안 뜨거나 enum 관련 에러로 죽음 | 로컬 PG를 안 띄웠거나 datasource가 H2를 보고 있다. 3번대로 `docker compose up -d postgres redis` 후 4번의 `application-local.yml`이 그 PG를 가리키는지 확인. |
| 로그인/POST 요청이 `403`(CSRF) | axios(`withCredentials: true`)는 `XSRF-TOKEN` 쿠키를 자동으로 `X-XSRF-TOKEN` 헤더에 실어 보낸다. 이 쿠키는 첫 GET 요청 응답에서 `CsrfCookieFilter`가 심어주므로, 페이지 최초 진입(첫 GET)이 없었거나 쿠키가 만료된 상태로 바로 POST를 날리면 403이 난다 — 페이지를 한 번 새로고침 후 재시도. |
| 로컬에서 카카오/구글 로그인 버튼이 안 됨 | OAuth client-id/secret이 로컬 `.env`에 없어서다. 로컬 개발은 소셜 대신 6번의 기업(자체) 로그인으로 진행 — 소셜은 배포 환경에서만 확인. |
| `npm run dev`로 로그인해도 실 백엔드 데이터가 안 보임 | MSW가 여전히 켜져 있다. 5번대로 `enableMocking()`을 임시로 주석 처리했는지 확인(원복 잊지 말 것). |
