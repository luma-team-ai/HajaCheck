## 1. 개요 (Overview)

건물 외벽·교량 등 시설물을 촬영한 **이미지/영상을 웹 플랫폼에 업로드**하면, AI(딥러닝 비전 모델)가 균열·박리·누수·철근 노출 등 하자를 자동 탐지하고, LLM(LangChain + RAG)이 점검 보고서 초안을 자동 생성해 주는 시설물 안전점검 지원 플랫폼.

### 1.1 배경 및 문제 정의

- 100m 이상 고층 건물 외벽, 고속도로 교량 하부 등은 사람이 직접 점검하기에 **추락 위험, 고비용, 장시간 소요** 등 부담이 크다.
- 육안 점검은 점검자 숙련도에 따라 판독 편차가 크고, 점검 이력이 체계적으로 축적되지 않는다.
- 국내(뷰메진 VODA, 드론와이드샷)·해외(Inspekt AI, Hammer Missions) 모두 촬영 이미지 기반 AI 시설물 점검 서비스가 상용화 단계에 있어 시장성이 검증된 주제다.

### 1.2 솔루션 요약

1. 촬영된 시설물 이미지/영상 **업로드** (촬영 수단 무관)
2. AI 모델(FastAPI 서빙)이 하자 **자동 탐지·분류·심각도 평가**
3. 결과를 웹 대시보드에서 **시각화**(원본 위 바운딩박스/세그멘테이션 오버레이)
4. LangChain 기반 LLM + RAG(점검 법규·기준 VectorDB)로 **점검 보고서 초안 자동 생성**
5. 점검 이력 축적 및 이전 점검 대비 변화 추적
6. **시나리오 챗봇 + 상담원 연결**로 사용 문의·점검 상담 지원

## 2. 목표 (Goals)

### 2.1 프로젝트 목표

- 4주 내 업로드→탐지→시각화→보고서 생성의 **End-to-End 데모 완성**
- 하자 탐지 모델 성능 목표: 균열 탐지 기준 **mAP@0.5 ≥ 0.7** (공개 데이터셋 기준, 팀 협의로 조정)
- 보고서 초안 생성 소요: 이미지 100장 기준 **10분 이내**

### 2.2 목표 사용자 (Persona)

| 페르소나 | 설명 | 핵심 니즈 |
|---|---|---|
| 안전점검 기술자 | 시설물 안전점검 업체 소속, 촬영 결과 판독 | 판독 시간 단축, 누락 방지 |
| 시설물 관리자 | 건물주·관리사무소·도로공사 담당자 | 상태 한눈에 파악, 이력 관리 |
| 점검 책임자 | 보고서 결재·제출 담당 | 법규 기준에 맞는 보고서 신속 작성 |
| 서비스 관리자 | 플랫폼 운영자 | 사용자·콘텐츠·상담 관리 |

### 2.3 범위 정의

**In-Scope (P0 — 필수)**

- 랜딩 페이지(로그인 전 — 서비스 소개·요금제·시작하기)
- 소셜 로그인(Kakao/Google OAuth2) + 마이페이지
- 시설물(프로젝트) 관리
- 이미지·영상 업로드 및 전처리(영상→프레임 추출)
- AI 하자 탐지: 균열(crack) 중심 + 박리·박락, 누수·백태, 철근 노출 등 다중 클래스
- 탐지 결과 시각화(오버레이, 확대, 하자별 필터) 및 수동 검수
- 하자 심각도 등급 분류
- LLM 점검 보고서 초안 자동 생성(LangChain) + PDF 내보내기
- RAG 기반 점검 기준 Q&A 챗봇 (LangChain + Chroma)
- 시나리오 챗봇 + 상담원 연결(실시간 채팅)
- 관리자 페이지(사용자·하자 기준·상담 관리)

**P1 — 여유 시 추가**

- 점검 이력 비교(동일 시설물 회차 간 하자 변화 추적)
- 하자 통계 대시보드(유형별/등급별/구역별)
- 균열 폭·길이 정량 추정(픽셀→실측 환산)
- **점검 계획·주기 관리**(FMS 패턴): 시설물별 점검 주기 설정, 다음 점검일 도래 대시보드 표시
- **지도 뷰**: 시설물을 지도(Kakao Map)에 마커 표시, 하자 최고 등급별 색상(녹/황/적) — 업로드 시 수집한 EXIF GPS 활용
- **인앱 알림 센터**: 헤더 벨 아이콘 + 알림 목록 — 분석 완료, 검수 대기, 상담 답변, 점검일 도래(주기 연계). 인앱 폴링 기반, 이메일·푸시는 제외
- **멤버십 쿼터**: 플랜별 사용량 제한(월 분석 장수 등) 체크 + 요금제 소개 페이지 + 마이페이지 "내 플랜" (§2.4)
- **메뉴별 경량 AI 기능** (전원 AI 구현 경험 목적 — 각 프롬프트 체인 1개 수준): 대시보드 AI 주간 브리핑, 하자 AI 설명 패널, 하자 자연어 검색 — **Standard 이상 플랜 전용**(§2.4 프리미엄 전략)

**P2 — 확장 목표 (Stretch)**

- **불법 증축 탐지**: 건축물대장/기준 형상 대비 외형 변화 탐지
- 외벽 3D/파노라마 매핑, 열화상 이미지 분석
- 체크리스트 기반 점검(사전점검 앱 패턴 — AI 중심 흐름과 결이 달라 참고 수준)
- **조치 보드(칸반)**: 하자 상태머신을 칸반 보드로 시각화, 드래그로 상태 전이

**Out-of-Scope**

- 촬영 장비(드론 등) 연동·제어
- 법적 효력을 갖는 정식 안전진단 판정(본 시스템은 **점검 보조 도구**)
- **PG 결제 연동** — 멤버십은 요금제 구조·쿼터까지만 구현, 결제는 "업그레이드 문의" 버튼으로 대체

### 2.4 비즈니스 모델 (멤버십·구독)

**목표 고객(B2B)**: 시설물 안전점검 업체, 건물·자산 관리업체, 지자체 시설 관리 부서

**설계 원칙 — 역할과 플랜의 축 분리**: 역할(Role)은 "무엇을 할 수 있는가"(권한, FR-1), 플랜은 "얼마나·누구와 쓸 수 있는가"(용량·좌석·프리미엄 기능). 두 축은 겹치지 않으며, **좌석(점검자 초대 수)이 두 축을 잇는 다리** — Free는 1인 체험, Standard부터 팀 협업(검수 워크플로)이 열린다.

**요금제(안)** — 과금 축: 시설물 수 + 월 분석 이미지 수 + 점검자 좌석 (실원가인 HF API 토큰·스토리지·분석 컴퓨팅과 비례)

| 플랜 | 시설물 / 월 분석 | 점검자 좌석 | 보고서 | 상담 | AI 부가 기능* | 대상 |
|---|---|---|---|---|---|---|
| Free | 1개 / 50장 | ✕ (1인 계정) | 워터마크 | 시나리오 챗봇만 | ✕ | 체험·개인 |
| Standard (월 구독) | 10개 / 1,000장 | 3명 | 정식 PDF | 상담원 연결 | ⭕ | 소규모 점검 업체 |
| Enterprise (별도 견적) | 무제한 / 협의 | 무제한 | + 데이터 반출 | 우선 응대 | ⭕ | 관리업체·공공 |

*AI 부가 기능 = P1 경량 AI(하자 설명·자연어 검색·주간 브리핑)와 회차 비교 — "핵심 AI 탐지는 전 플랜, AI 편의 기능은 유료" 프리미엄 전략(HF 토큰 원가와 비례)

**조직 모델(단순화)**: 별도 조직(워크스페이스) 테이블 없이 **계정 소유자 = 플랜 보유자, 소유자가 점검자를 좌석 한도 내 초대** — 4주 범위에 맞는 최소 모델

**구현 범위**: 플랜 테이블·사용량 쿼터 체크(`QuotaInterceptor` — 역할 체크와 분리)·좌석 관리·요금제 페이지·마이페이지 플랜 표시까지 P1, 쿼터/좌석 초과 시 업그레이드 안내 모달 표준 UX. PG 결제는 Out-of-Scope

**사업 목표(가설)**: 점검 1건당 판독 인건비 대비 절감 효과를 근거로 Standard 전환 유도 — 완료보고 시 원가·손익 시뮬레이션 1장 포함

## 3. 유사 플랫폼 벤치마킹

| 플랫폼 | 국가 | 참고 포인트 |
|---|---|---|
| 뷰메진 VODA | 국내 | 외벽 이미지 기반 2D/3D 가상화 + AI 자동 결함 탐지 |
| 드론와이드샷 | 국내 | 하자 유형 분류 + 분석 보고서 제공 서비스 모델 |
| Inspekt AI | 해외 | 대시보드에서 자산 상태 평가, 회차별 비교 기능 |
| Hammer Missions | 해외 | 탐지 결과에 어노테이션 추가·팀 공유, 보고서 자동 생성 |
| FMS(시설물통합정보관리시스템) | 국내 공공 | 시설물 제원·이력 → **점검계획 수립** → 진단 → 보수·보강의 표준 업무 흐름 (점검 주기 관리 P1의 근거) |
| 자이홈·채들 등 하자접수 앱 | 국내 | 하자 **접수→처리중→보수완료 상태 추적** + 관리자 실시간 공유 (하자 상태머신의 근거) |

공통 패턴: **자산(시설물) 단위 관리 → 점검 회차 → 이미지 뷰어(오버레이) → 결함 목록 → 자동 보고서** 흐름. hajaCheck 메뉴 구성도 이 흐름을 따른다.

## 4. 메뉴 구성 (IA)

```
🏠 랜딩 페이지 (로그인 전)
 └ 서비스 소개, 핵심 기능, 요금제(§2.4), 시작하기(로그인 유도)

🔔 알림 센터 (P1 — 전역 헤더 벨 아이콘)
 └ 분석 완료 / 검수 대기 / 상담 답변 / 점검일 도래 — 인앱 폴링, 읽음 처리

🔐 로그인/회원가입
 └ Kakao / Google 소셜 로그인

📊 대시보드
 ├ 전체 시설물 현황, 최근 점검 요약, 하자 등급 분포, 처리 대기 현황(검수 대기·조치 대기 건수)
 ├ 다음 점검일 도래 시설물 표시 (P1 — 점검 주기 연계)
 └ AI 주간 브리핑 카드 (P1 — LLM 자연어 요약)

🏢 시설물 관리
 ├ 시설물 목록/등록 (건물·교량, 위치, 규모, 준공연도)
 ├ 시설물 상세 (기본정보, 점검 이력 타임라인)
 │   └ 회차 간 하자 변화 비교 (P1 — 이전 점검 대비 추적)
 ├ 점검 주기 설정 (P1 — FMS 패턴, 다음 점검일 자동 계산)
 └ 지도 뷰 (P1 — EXIF GPS 기반 마커, 하자 최고 등급별 색상)

🔍 점검 관리
 ├ 점검(회차) 생성 — 시설물 선택, 점검일, 담당자
 ├ 촬영 데이터 업로드 — 이미지/영상 일괄 업로드, 진행률 표시
 ├ AI 분석 실행/상태 — 분석 큐, 진행 상황
 ├ 분석 결과 뷰어 ★핵심 화면
 │   · 원본 이미지 + 하자 오버레이(바운딩박스/마스크)
 │   · 하자 유형·등급 필터, 확대/이동
 │   · 탐지 결과 수동 확인·수정(오탐 제거, 등급 조정)
 │   · AI 하자 설명 패널 (P1 — 선택 하자 원인·조치 LLM 생성)
 └ 보고서 생성 진입점 — 회차 상세에서 바로 생성 (벤치마킹 표준 흐름: 점검 회차의 산출물)

📋 하자 관리
 ├ 하자 목록 (유형/등급/상태별 필터·검색)
 ├ 자연어 검색 (P1 — 질의를 LLM이 필터 조건으로 변환, 데이터 질의 단일 창구)
 ├ 조치 보드 (P2 — 칸반, 드래그로 상태 전이)
 └ 하자 상세 (이미지, 위치, 등급, 조치 상태, 메모)
     · 상태머신: 신규(AI탐지) → 검수확정 → 조치대기 → 조치중 → 조치완료 (하자접수 앱 패턴)

📄 보고서 (전체 이력·편집 중심 — 생성은 점검 회차 상세에서)
 ├ 보고서 목록 / 이력 관리
 ├ 보고서 편집·미리보기
 └ PDF 내보내기

💬 고객지원
 ├ AI 어시스턴트 — 점검 기준·법규 Q&A (RAG 전담. 데이터 질의는 하자 관리 자연어 검색으로 일원화)
 ├ 상담 챗봇 — 시나리오 기반 안내(버튼 플로우) → 상담원 연결(실시간 채팅)
 │   · 진입: 전역 플로팅 버튼(우하단) — 메뉴는 이력 조회 중심
 └ 내 상담 이력

👤 마이페이지
 ├ 내 정보 (프로필, 연동 계정 관리)
 ├ 내 점검 이력 / 내 보고서
 ├ 내 플랜 (P1 — 멤버십 플랜·사용량 표시, 업그레이드 문의)
 └ 내 상담 내역

📈 통계 (P1)
 └ 시설물별·기간별 하자 추이, 회차 간 비교

🛠 관리자 페이지 (관리자 전용)
 ├ 사용자 관리 (회원 목록, 권한 변경, 정지)
 ├ 플랜·쿼터 관리 (P1 — 사용자별 플랜 변경, 사용량 조회, §2.4)
 ├ 하자 유형·등급 기준 관리
 ├ 상담 관리 (상담 대기열, 상담원 채팅 콘솔, 챗봇 시나리오 관리)
 ├ RAG 문서 관리 (법규·지침 업로드/재임베딩 배치)
 ├ 서비스 통계 (가입자, 분석 건수, 상담 건수)
 └ 시스템 모니터링 (헬스체크, 분석 잡 큐, 에러 로그, 서버 자원, HF 사용량·예산 가드레일)
```

## 5. 핵심 기능 요구사항

### FR-1 소셜 로그인 & 마이페이지

- Kakao / Google OAuth2 로그인 (Spring Security OAuth2 Client)
- 세션 관리: Spring Session + Redis
- 마이페이지: 프로필 조회·수정, 소셜 계정 연동 관리, 내 점검 이력·보고서·상담 내역 조회, **내 플랜·사용량 표시 + 점검자 좌석 초대 관리(P1, §2.4)**
- 역할(Role): 일반 사용자 / 점검자 / 관리자 / 상담원 — 계층: `ADMIN > INSPECTOR > USER`, `COUNSELOR`는 별도 축(관리자 겸임 가능)
- **역할별 권한 매트릭스**:

| 기능 | 일반 사용자 | 점검자 | 상담원 | 관리자 |
|---|---|---|---|---|
| 시설물 등록·조회 | 본인 소유만 | 배정 시설물 | ✕ | 전체 |
| 업로드·분석 요청 | ⭕ (플랜 쿼터 내) | ⭕ | ✕ | ⭕ |
| **검수(오탐 수정·등급 확정)** | ✕ | ⭕ 핵심 권한 | ✕ | ⭕ |
| 하자 상태 전이 | 조치 완료 보고만 | ⭕ | ✕ | ⭕ |
| 보고서 생성·편집 | 열람·다운로드만 | ⭕ | ✕ | ⭕ |
| 상담원 콘솔 | ✕ | ✕ | ⭕ 전용 | ⭕ |
| 관리자 페이지 | ✕ | ✕ | 상담 관리만 | ⭕ 전체 |

- **권한 집행 원칙**: 프론트 메뉴 숨김(`ProtectedRoute`/`AdminRoute`)은 편의일 뿐 — 백엔드에서 `@PreAuthorize`(역할) + 소유권 데이터 필터를 반드시 병행. 플랜 제한은 별도 `QuotaInterceptor`로 분리(§2.4)
- **CSRF 방어**: 쿠키 세션 기반이므로 상태 변경 요청(POST/PUT/PATCH/DELETE)에 Spring Security CSRF 토큰 적용 — React는 `XSRF-TOKEN` 쿠키를 읽어 헤더로 전송

### FR-2 촬영 데이터 업로드

- 이미지(JPG/PNG) 다중 업로드, 영상(MP4) 업로드 시 프레임 자동 추출(간격 설정)
- 대용량 대비 청크 업로드·진행률 표시, 업로드 완료 시 썸네일 생성
- 메타데이터 추출(EXIF: 촬영시각, GPS 좌표 — 위치 매핑에 활용)
- **업로드 보안 검증**: 확장자 외 **매직바이트(파일 시그니처) 검증** + 용량 상한, 원본은 직접 서빙하지 않고 서버 측 재인코딩본(썸네일·뷰어용)만 제공

### FR-3 AI 하자 탐지

- 탐지 클래스(안): 균열, 박리·박락, 누수·백태, 철근 노출, 도장 손상 (데이터셋 사정에 따라 조정)
- 모델: **YOLOv8-seg(인스턴스 세그멘테이션)** — 균열은 마스크, 면적형 하자는 박스로 단일 모델 커버. P1 균열 폭·길이 정량 추정의 기반. AI Hub「건물 균열 탐지 이미지」등 공개 데이터셋 활용, 학습은 Colab 등 GPU 환경에서 수행 후 서버에는 추론만 배포
- **ARM CPU 추론 최적화**: 학습 산출물에 ONNX 변환(+INT8 양자화) 파이프라인 포함 — ONNX Runtime 서빙으로 순정 PyTorch 대비 2~4배 속도 확보(100장 10분 목표의 전제). 최적화를 후반으로 미루지 않고 1차 학습부터 적용
- **모델 버전 관리**: 파일명 `{날짜}_{데이터셋버전}_{mAP}` 규칙 + `models/MODEL_CARD.md`(학습 데이터·하이퍼파라미터·성능·한계) 유지 — 배포 중 모델 추적 가능하게
- Spring Boot → FastAPI 비동기 추론 요청, 결과(클래스, 좌표, confidence)를 PostgreSQL 저장
- 심각도 등급: 규칙 기반(크기·유형) 1차 → 점검자 수동 조정

### FR-4 결과 시각화·검수

- 이미지 뷰어에 탐지 결과 오버레이, confidence 임계값 슬라이더
- 점검자가 오탐 삭제/누락 추가/등급 수정 가능(휴먼 인 더 루프)
- **수정 이력 구조화**: `defects` 직접 UPDATE 대신 `defect_revisions` 테이블에 append-only 기록(누가·언제·무엇을 변경) — 검수의 감사 추적 보장, 삭제는 Soft Delete
- **하자 상태머신**: `신규(AI탐지) → 검수확정 → 조치대기 → 조치중 → 조치완료` 표준 전이만 허용(역방향·건너뛰기는 사유 기록 필수)

### FR-5 LLM 보고서 생성 (LangChain)

- LLM: **Qwen3-8B (Apache 2.0) — HuggingFace Serverless Inference API** 호출, GPU 확보 시 Ollama 로컬 서빙 전환 가능(LangChain 추상화로 설정 변경만으로 교체)
- LangChain 체인 구성: 시설물 정보 + 확정 하자 목록·통계 → 보고서 생성 프롬프트 → LLM
- **템플릿 채우기 방식**: 자유 서술이 아닌 섹션별 짧은 생성 → 조립 구조로 sLLM 품질 편차 방어
- RAG: 점검 지침·법규 문서(Chroma Retriever) 근거 인용
- **Grounding Check(사실 검증)**: 생성 체인 마지막에 후처리 검증 단계 — LLM이 언급한 하자 개수·등급을 `defects` 실측치와 코드로 대조, 불일치 시 해당 섹션 재생성 또는 경고 배지 표시(sLLM 수치 환각 방어)
- **섹션 병렬 생성**: 독립 섹션(개요/요약/상세/권고)은 `RunnableParallel`로 동시 호출 — 순차 대비 지연시간 단축(100장 10분 목표 대응)
- 출력: 점검 개요, 하자 현황 요약표, 유형별 상세, 조치 권고, 종합 의견 → 편집 후 PDF 내보내기

### FR-6 RAG 챗봇 (LangChain)

- 시설물안전법·안전점검 세부지침 등 문서 임베딩(Chroma), LangChain RetrievalQA 파이프라인
- 점검 기준·법규 질의 **전담** (데이터 질의는 하자 관리 "자연어 검색"으로 일원화 — 기능 중복 방지), 답변 출처 표시
- 대화 이력 관리(LangChain Memory + Redis)
- **LLM 응답 캐시**: 프롬프트 해시 키 기반 Redis 캐시 — 개발·테스트 중 동일 질의 반복 호출로 인한 HF 크레딧 소진 방지

### FR-7 상담 챗봇 & 상담원 연결

- **시나리오 챗봇**: 버튼 기반 플로우(예: 서비스 이용 방법 / 분석 결과 문의 / 요금·기타) — 시나리오는 관리자 페이지에서 등록·수정. 진입은 **전역 플로팅 버튼(우하단)**, 고객지원 메뉴는 이력 조회 중심
- **상담원 연결**: 시나리오에서 해결 안 될 경우 "상담원 연결" → 상담 대기열 등록 → 상담원(관리자 페이지 콘솔)과 **실시간 1:1 채팅(WebSocket/STOMP)**
- 상담 종료 시 대화 내역 저장, 마이페이지에서 조회 가능
- 상담원 부재 시: 대기 순번 안내 + 문의 남기기(오프라인 티켓)
- **WebSocket 인증**: STOMP `CONNECT` 프레임에서 `ChannelInterceptor`로 Spring Session 검증 — 세션 없는 연결 즉시 거부(HTTP 인증 미들웨어가 WS에 자동 적용되지 않는 누락 지점)

### FR-8 관리자 페이지

- 사용자 관리: 목록·검색, 권한 변경, 계정 정지
- **플랜·쿼터 관리(P1)**: 사용자별 플랜 조회·변경, 사용량(월 분석 장수) 모니터링, 업그레이드 문의 처리 (§2.4)
- 하자 유형·등급 기준 관리(코드 테이블)
- 상담 관리: 대기열 모니터링, 상담원 채팅 콘솔, 챗봇 시나리오 CRUD
- RAG 문서 관리: 법규·지침 PDF 업로드 → 임베딩 파이프라인 실행 — **재임베딩은 명시적 배치 잡으로 분리**(Chroma 임베디드 쓰기 락이 서비스 중 챗봇 질의를 막지 않도록)
- 서비스 통계: 가입자 수, 분석 요청 수, 상담 건수
- **시스템 모니터링** (간단 대시보드):
  - 서비스 헬스체크: Spring Boot Actuator(`/actuator/health`) + FastAPI(`/health`) 상태를 주기 폴링, 정상/이상 상태 표시
  - 분석 잡 큐 현황: 대기/진행/완료/실패 건수 (Redis 잡 상태 집계), 실패 잡 재실행 버튼
  - 최근 에러 로그: API 5xx·분석 실패 최근 N건 조회
  - 서버 자원: CPU·메모리·디스크 사용률 (Actuator metrics 활용)
  - HF Inference API 호출량·누적 토큰 카운트 (예산 관리용)
  - **예산 가드레일**: 일일/주간 토큰 한도 초과 시 관리자 경고 배지 + 초과 심화 시 디그레이드 모드(챗봇이 LLM 생성 생략, 문서 검색 결과만 반환)

### FR-9 알림 센터 (P1)

- 헤더 벨 아이콘 + 알림 드롭다운/목록, 미읽음 뱃지, 읽음 처리
- 알림 이벤트: 분석 완료, 검수 대기 발생, 상담 답변 도착, 점검일 도래(점검 주기 연계)
- 구현: `notifications` 테이블 + 폴링(30초) — WebSocket 푸시·이메일은 범위 제외(상담용 WS와 별개로 단순 유지)
- 이벤트 발행은 각 도메인 서비스에서 공통 `NotificationService` 호출로 통일

## 6. 시스템 아키텍처

**아키텍처 결정: 모놀리식** (Spring Boot 단일 앱 + FastAPI). JEUS 미사용 — Nginx가 Spring Boot 내장 Tomcat으로 직접 프록시.

```
[React SPA (Axios, WebSocket)]
      │ HTTPS / WSS
[Nginx (80/443)]
  ├─ React 정적 빌드 서빙 (/)
  ├─ /api/** ─▶ Spring Boot:8080 (리버스 프록시)
  └─ /ws/**  ─▶ Spring Boot:8080 (WebSocket 업그레이드 프록시)
      │
[Spring Boot 단일 앱 (JDK17, 내장 Tomcat, 8080)]
  ├─ 인증: OAuth2 (Kakao/Google) + Spring Session ─ Redis
  ├─ 실시간 상담: WebSocket/STOMP (상담원 연결)
  ├─ 업무 데이터 ─ PostgreSQL (사용자, 시설물, 점검, 하자, 보고서, 상담)
  └─ 추론/생성 요청 ─▶ [FastAPI (Python, 8000 — 내부 네트워크 전용)]
                        ├─ DL 모델 (하자 탐지: YOLOv8-seg)
                        ├─ LangChain (보고서 생성 체인, RAG 챗봇 파이프라인)
                        ├─ LLM: Qwen3-8B — HuggingFace Serverless Inference API
                        │    (종량제, LangChain HuggingFaceEndpoint 연동)
                        │    └ (대체) 동일 모델 Ollama 로컬 서빙 — GPU 확보 시 전환
                        └─ Chroma VectorDB — FastAPI 내 임베디드 모드(로컬 파일 저장)
                             임베딩: HuggingFace 한국어 모델(ko-sbert / BGE-m3, CPU — 무료)
[파일 스토리지] 업로드 원본·썸네일·오버레이 이미지
```

- **모듈러 모놀리스**: Spring Boot 내부 패키지를 도메인별(auth / core / counsel / admin)로 엄격 분리 — 향후 서비스 분리 가능한 구조로 설계
- **ORM: Spring Data JPA(Hibernate)** — Entity·Repository 규약은 `SpringBoot_코드_컨벤션.md` 참조. 프론트 규약은 `React_코드_컨벤션.md` 참조
- Spring Boot ↔ FastAPI: REST + 비동기 잡 패턴(분석 요청 → 잡 ID → 폴링/콜백). FastAPI·DB·Redis·Chroma는 외부 포트 미개방(내부 통신 전용)
- **FastAPI 워커 분리**: CPU-bound인 YOLO 추론은 별도 워커 프로세스(`ProcessPoolExecutor` 또는 uvicorn 워커 분배)로 오프로드 — 분석 진행 중에도 챗봇·보고서 API가 블로킹되지 않게(4 OCPU: 탐지용 1~2 + API용 1~2)
- **Correlation ID**: 모든 분석 잡·요청에 `X-Request-Id` 발급, Spring·FastAPI 양쪽 로그에 동일 ID 기록 — 경계 넘는 요청 추적(디버깅 시간 절감)
- 영상 프레임 추출: FastAPI 측 OpenCV/ffmpeg
- **Redis 활용처**: ① Spring Session(로그인 세션) ② 분석 잡 상태·진행률 캐시 ③ LangChain 챗봇 대화 메모리 ④ 상담 대기열(Sorted Set) ⑤ 대시보드 통계 캐시 ⑥ OAuth 인가 state 임시 저장 ⑦ LLM 응답 캐시(프롬프트 해시 키)

### 6.1 배포 환경 — OCI (Oracle Cloud Infrastructure)

**인프라: OCI Always Free — Ampere A1 VM 1대 (ARM, 4 OCPU / 24GB RAM, Ubuntu 22.04)**

```
[도메인 DNS] ── A레코드 ──▶ [OCI Reserved Public IP(고정)]
                                    │
[OCI VCN — 퍼블릭 서브넷, Security List: 80/443/22 전체 허용 + 5432/6379는 팀원 IP만]
 └─ Ampere A1 VM
     ├─ Nginx (80/443) — React 정적 서빙, 리버스 프록시, certbot(Let's Encrypt)
     ├─ Spring Boot jar (8080, systemd)
     ├─ FastAPI/uvicorn (8000, systemd) — YOLOv8-seg 추론(CPU) + Chroma 임베디드
     └─ PostgreSQL / Redis — apt 직접 설치, systemd (외부 포트 미개방)
```

- **서버에는 Docker 미사용**: 전 구성요소 직접 설치 + systemd 관리. Chroma는 FastAPI 임베디드 모드로 별도 프로세스 없음 → ARM 이미지 호환 문제 원천 회피, 관리 단순화

**개발 환경 — 공용 개발 DB (공인 IP 직접 접속)**

- OCI 서버의 PostgreSQL(5432)·Redis(6379)를 **팀 공용 개발 DB**로 사용. 포트를 개방하고 팀원들이 공인 IP로 직접 접속 — 로컬 Docker 불필요, `application-local.yml`에 서버 IP만 설정
- **보안 필수 조치** (포트 개방의 전제조건):
  - OCI Security List 인그레스를 0.0.0.0/0이 아닌 **팀원 IP 대역만 허용**(IP 변동 시 콘솔에서 갱신)
  - PostgreSQL: 강력한 비밀번호 + `pg_hba.conf`에서 접속 IP 제한
  - Redis: `requirepass` 필수 설정(미설정 시 인증 없이 전체 접근 가능), bind 설정 확인
- **개발/운영 분리**: PostgreSQL은 `hajacheck_dev` / `hajacheck_prod` 데이터베이스 분리, Redis는 DB 인덱스 분리(운영 0번 / 개발 1번)
- 팀 규칙: 공용 dev DB의 스키마 변경(DDL)은 사전 공지 후 진행. **개발 DB에 실사용자 데이터 저장 금지** — 시연·테스트용 합성 데이터만 사용(개인정보가 팀원 전원의 로컬 접속 구간에 노출되는 것 방지)
- 저장소의 `docker-compose.yml`(PostgreSQL+Redis)은 예비용으로 유지: ① 오프라인·서버 장애 시 단독 작업 ② 파괴적 테스트(스키마 실험 등) 격리 실행
- 방화벽 이중 설정 주의: OCI Security List + VM 내부(ufw/iptables) 모두 개방 필요
- Nginx WebSocket 프록시 설정(`Upgrade`/`Connection` 헤더) 필수 — 상담 기능 전제조건
- 보안: SSH 키 인증, 80/443은 전체 개방, 5432/6379는 팀원 IP 대역만 허용, FastAPI(8000)는 외부 접근 차단
- LLM(HF Serverless API)은 외부 API 호출이라 서버 사양 무관, YOLO 추론은 ARM CPU로 수행(데모 수준 충분)

**CI/CD: GitHub Actions**

- CI (PR 시): Spring Boot 빌드+테스트, FastAPI 테스트, React 빌드 — path filter로 변경 파트만 실행
- CD (main 머지 시): 빌드 산출물(jar, React dist)을 SSH/scp로 OCI VM 전송 → systemd 서비스 재시작
- SSH 키·서버 IP 등은 GitHub Secrets 관리

**리스크 대비**

- A1 무료 인스턴스는 리전 용량 부족("Out of capacity")이 잦음 — **착수 주간(7/9~13)에 선확보**, 실패 시 PAYG 전환(Always Free 한도 내 과금 없음)
- **시연 백업 플랜**: 완료보고(8/7) 당일 로컬 실행 환경 별도 준비

### 6.2 AI 개발 컨벤션 (전원 AI 구현 체제의 품질 통일 장치)

> 상세 실무 규약은 별도 문서 **`AI_개발_컨벤션.md`** 참조 (디렉토리 구조, 코드 예시, 개발 절차, 리뷰 체크리스트 포함)

각 메뉴 담당이 AI 기능을 직접 구현하되, 아래 공통 기반 위에서만 개발한다. AI-LLM 코치가 1주차에 공통 모듈·컨벤션 문서·예시 체인 1개를 작성해 온보딩 세션(7/15)에서 배포하고, 코드 리뷰 ①(7/17)에서 준수 여부를 점검한다.

1. **공통 LLM 클라이언트 단일화**: FastAPI에 `llm_client` 공통 모듈(모델명·HF 엔드포인트·타임아웃·재시도·토큰 사용량 로깅 일원화). 모든 체인은 이 클라이언트만 사용 — 모델 교체(HF API↔Ollama) 시 이 파일만 수정
2. **프롬프트 파일 분리**: 하드코딩 금지, `prompts/` 디렉토리에 파일 관리(`report_summary.md`, `defect_explain.md` 등), 공통 시스템 프롬프트 템플릿 사용
3. **출력 형식 통일**: LLM 응답은 JSON 스키마 지정(structured output)으로만 수신 — 자유 텍스트 파싱 금지
4. **엔드포인트·에러 규약**: AI 엔드포인트 네이밍 통일(`/ai/report`, `/ai/chat`, `/ai/nl-search` 등), 요청/응답 공통 포맷, 실패 시 표준 폴백 처리
5. **버전 고정**: LangChain·chromadb 등 requirements 버전 고정 (LangChain은 버전 간 API 변화가 큼)
6. **법규 문서 청킹**: 임의 글자수 분할 금지 — 법조문 경계 우선 분리자 사용(`RecursiveCharacterTextSplitter(separators=["\n제", "\n①", "\n\n", "\n"])`) — 조문 중간 절단은 출처 인용 정확도(KPI) 저하 직결

> DL 추론은 AI-DL 담당의 단일 엔드포인트를 호출하는 구조라 별도 컨벤션 불필요.
