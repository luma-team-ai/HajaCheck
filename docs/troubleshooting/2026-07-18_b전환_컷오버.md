## 1. [P1·차단] main 머지 = 자동배포 자동 발화 → 어중간한 실패 위험

**상황**: dev→main 머지 시 `cd.yml`(`push: [main]` → SSH `deploy/deploy.sh`)이 **자동으로** 서버 배포를 실행한다. 새 compose는 redis 6380 충돌 + 빈 DB로 실패하는데, 그 사이 구 컨테이너가 재생성되다 멈추면 프로덕션이 다운된 채 방치된다(arm1은 자동 롤백 없음).

**원인**: cd.yml이 승격(머지)과 배포를 커플링. 컷오버는 수동 순서(구 redis 정지 → dump → restore → deploy)가 필요한데 자동배포가 그보다 먼저 터짐.

**해결**: 머지 **전** `gh workflow disable cd.yml`로 자동배포 잠금 → 수동 컷오버 → 완료 후 `gh workflow enable cd.yml`. (워크플로우 파일 수정 아님 = PR머신 repo 무수정 원칙 유지.)

**교훈**: 컷오버·인프라 전환을 동반한 승격은 **항상 cd.yml을 먼저 disable**하고 수동 진행. 일반 승격(코드만)은 그대로 자동배포 OK.

## 2. [P1·차단] 스키마 드리프트 — Flyway 부재로 spring validate 기동 실패

**상황**: 컷오버 중 데이터 restore 후 새 spring이 안 뜨고 재기동 반복.
```
Schema-validation: missing column [assigned_inspector_id] in table [inspections]
```

**원인 (근본)**: 프로젝트에 **Flyway가 없다**. arm1은 `ddl-auto=validate`라 스키마가 미리 정확히 있어야 뜨는데, 복원한 옛 prod 스키마엔 새 코드가 기대하는 컬럼이 없었다. 로컬은 `ddl-auto=update`라 개발자마다 자동 생성돼 **로컬에선 안 보이고 운영에서만 터지는** 구조적 드리프트.

**해결 (임시)**: 새 postgres 컨테이너에서 `SPRING_JPA_HIBERNATE_DDL_AUTO=update`로 **1회만** 부팅 → Hibernate가 누락 컬럼 자동 추가(데이터 무손실 확인: users 7·companies 2·user_consents 4 유지) → override 제거 후 `validate`로 재기동 → 정상. 원본 host DB는 안 건드리고 복원 복사본에서만 작업.

**교훈**: **다음 승격에도 재발한다** → **#359(P1) Flyway 도입 필수**(현 운영 스키마를 `V1__baseline.sql`로, 이번 update 델타 포함). 그전까지 승격은 반드시 컷오버 런북으로 수동 진행하며 "빈 DB→validate 기동실패"를 예상할 것. **ddl-auto=update 임시 봉합은 마이그레이션 기록이 안 남으므로** 반드시 Flyway로 정본화.

## 3. [경미] pg_restore CAST 에러 4건 무시됨 — 무해

**상황**: `pg_restore` 중 `ERROR: could not execute query: CREATE CAST (??? AS character varying) ...` 4건, "errors ignored on restore: 4".

**원인**: 소스 DB의 커스텀 CAST 정의(타입명이 덤프 표시에서 깨져 `???`)가 복원 실패. 테이블·데이터와 무관.

**해결**: 무시하고 진행. **row 수가 baseline과 정확히 일치**(7/4/2)함을 확인해 데이터 온전성 검증. `--no-owner --no-privileges`로 역할 소유권 에러도 억제.

**교훈**: pg_restore는 무시 가능한 에러에도 exit=1을 낸다 → 성공 판정은 exit code가 아니라 **복원 후 row 수 대조**로.

## 4. [문서 오류] 런북 §2.5 "SSH 터널 5432→5433 공지"는 잘못된 전제

**상황**: 런북이 컷오버 후 "개발자 SSH 터널 포트 5432→5433 변경 공지"를 지시.

**원인**: "개발 DB도 컨테이너로 이전"을 전제했으나 실제 컷오버는 **prod(`hajacheck`)만 전용 컨테이너(5433)로 옮기고 개발용 `hajacheck_dev`는 구 host postgres(5432)에 그대로 남겼다**. 5433 컨테이너엔 prod DB만 있어, 개발자를 5433으로 보내면 **로컬 개발이 프로덕션 DB에 붙어 실데이터 오염**.

**해결**: 공지 문구를 "**로컬 개발 환경 변경 없음**(기존 5432 터널 + `hajacheck_dev` 유지)"로 정정. 런북 §2.5에 정정 주석 추가. host postgres(5432)는 pension_db·smartfarm도 얹혀 있어 내려가지도 않음.

**교훈**: 컷오버는 prod만 이전 = 개발자 워크플로우 무영향 → **별도 공지 불필요**. 5433은 arm1 서버 내부 전용(`127.0.0.1` 바인딩, 외부 미노출).
